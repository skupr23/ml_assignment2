# ml_assignment2
ჩვენი ამოცანა იყო დაგვეტრენინგებინა მოდელი რომელიც დატრენინგდებოდა დატასეტზე და გამოიცნობდა ტრანზაქცია fraud არის თუ არა. რადგან გვქონდა კლასიფიკაციის ამოცანა, გამოვიყენე 4 სხვადასხვა ტიპის მოდელი: logistic regression, decision tree, random forest და XGBoost.

feature engineering:

მონაცემები შედგება transaction და identity ფაილებისგან. ორივე train და test ნაწილში ისინი გაერთიანდა TransactionID-ის მიხედვით.

Target ცვლადი isFraud გამოიყო ცალკე, ხოლო TransactionID ამოღებულია თრეინინგიდან, რადგან ამ ცვლადს გავლენა არ აქვს ტრანზაქციის შინაარსზე და მხოლოდ გადანომრისთვისაა გამოყენებული. TransactionID შენარჩუნებულია მხოლოდ submission ფაილისთვის.

ნალ ველიუები ხელით არ შემივსია. მათი დამუშავება გადავიტანე Pipeline-ში, რათა training და inference ერთნაირად დამუშავდეს. რიცხვითი ცვლადებისთვის გამოვიყენე მედიანით შევსება, ხოლო კატეგორიული ცვლადებისთვის - ყველაზე ხშირი მნიშვნელობით შევსება.
გარდა ამისა, გამომდინარე იქედან რომ დატასეტში ბევრად მეტი სანდო ტრანზაქცია იყო ვიდრე fraud, დატასეტი დავაბალანსე და ამოვაკელი სანდო ტრანზაქციების ნაწილი. 


თავდაპირველად ბეიზ მოდელებისთვის შედარებით მარტივი ინჟინერია გამოვიყენე და მხოლოდ transacrionDTდან გამოვყავი ტრანზაქციის საათი და დღე, აგრეთვე თანხის ოდენობა გადავაქციე ლოგარითმულად რათა უფრო მცირე ზომის რიცხვებზე 
ჰქონოდა მოდელს სამუშაო. შემდგომში კი დავამატე ტრანზაქციის გამგზავნის და მიმღების მეილებიც და აგრეთვე ცვლადი ემთხვევა თუ არა ამ მეილების დომეინები. ამ ლოგიკით მოდელს შესაძლებობა მიეცა უფრო მეტი კანონზომიერება დაეჭირა იუზერების მეილებსა და ტრანზაქციის სანდოობაში. 

რაც შეეხება ენკოდირებას 
Logistic Regression-ისთვის გამოვიყენე One-Hot Encoding, რადგან ამ მოდელისთვის კატეგორიულ ცვლადებს შორის ხელოვნური რიცხვითი იერარქია არ უნდა შეიქმნას.
Decision Tree, Random Forest და XGBoost-ისთვის გამოვიყენე Ordinal Encoding, რადგან ხის მოდელები უკეთ იტანენ ამ ფორმატს და ეს მიდგომა მეხსიერების მხრივ უფრო მსუბუქია.

feature selection:

აქ გამოვიყენე მარტივი წესები: 
1. დავტოვე მხოლოდ ისეთ ცვლადები, სადაც ნალები არ შეადგენდნენ 90%.
2. კატეგორიული ცვლადებიდან დავტოვე მხოლოდ დაბალი cardinality-ის მქონე სვეტები.
3. კატეგორიული cardinality-ის ზღვარი იყო 50 უნიკალუტი მნიშვნელობა.
4. საბოლოოდ შევქმენი final_cols სია, რომელიც გამოიყენება Pipeline-ში.

ეს მიდგომა ამცირებს მეხსიერების პრობლემის და overfitting-ის რისკს.

ტრენინგი: 
Logistic Regression გამოვიყენე baseline მოდელად. ის მარტივი და ინტერპრეტირებადია, მაგრამ რთულ არაწრიფ მიმართებებს კარგად ვერ იჭერს. თრეინს და საბმიშენის ქულები ჰქონდა ყველაზე ცუდი ამ მოდელს, თუმცა პრობლემა ჰქონდა 
თავად მოდელს დატასეტის ზომასთან, რადგან ყველაზე კარგი ამ მოდელისთვის არის onehot კოდირება, რომელიც ძალიან ზრდის დატასეტის ზომას. ამის გამო მოდელი ხშირად ვერ convergedებოდა და მომიწია მაქსიმალური იტერაციების 
პარამეტრის ხელით გაზრდა, რამაც რადიკალურად გაზარდა ტრენინგის დრო. 

Decision Tree გამოვიყენე non-linear baseline-ად. ის სწრაფია და მარტივი გასაგებია, მაგრამ მაღალი რისკი აქვს ოვერფიტში წასვლის, განსაკუთრებით თუ max_depth არ არის შეზღუდული. მიუხედავად ამ პარამეტრის შეზღუდვისა ამ მოდელმა საბმიშენზე მაინც ცუდი შედეგი აჩვენა. 

Random Forest გამოვიყენე Decision Tree-ის უფრო სტაბილურ ალტერნატივად. რამდენიმე ხის გაერთიანება ამცირებს ვარუაციას, თუმცა ტრენინგის დრო და საჭირო მეხსიერების ზომა უფრო მაღალია.

XGBoost იყო ყველაზე ძლიერი კანდიდატი. კარგად იჭერს არაწრფივ დამოკიდებულებებს და აქვს ოვერფიტის კონტროლი ბევრად მარტივია პარამეტრი, როგორიცაა max_depth, learning_rate, subsample, colsample_bytree, min_child_weight და reg_lambda.
